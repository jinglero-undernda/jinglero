# BUG_XXXX – Title Goes Here

Copy this template into `BUG_XXXX.md`, replace placeholders, and keep the entry updated throughout the investigation.

## Metadata

- **Status**: `open | in progress | resolved`
- **Last Updated**: YYYY-MM-DD
- **Category**: e.g., `frontend`, `backend`, `auth`, `billing`, `infra`
- **Severity**: `blocker | high | medium | low`
- **Environment**: browser + version, OS, API env, feature flags, etc.
- **Related Commits / PRs**: list commit hashes or URLs (use messages like `BUG_0001: ...`)

## Reproduction Steps

1. Step-by-step instructions that consistently trigger the bug.
2. Include expected vs actual behavior notes where relevant.

## Description

Concise summary of the user-facing impact and any screenshots, logs, or clarifying notes.

## Root Cause

Document the confirmed underlying issue. Include hypotheses explored (and ruled out) if useful for future reference.

## Resolution

Outline the fix applied, validation performed (tests, manual verification, environments), and any follow-up tasks or monitoring items.

## Related Files

- `path/to/file`: search for `importantFunction` – note why this file matters.
- Add as many entries as needed, each with a quick search hint.

## Related Code

Short snippets or references (function names, components, endpoints) that were directly touched. Keep it terse; link to commits for full diffs.
